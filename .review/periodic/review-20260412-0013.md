## New Optimization Suggestions

### 1. External Process Architecture (from daniel3303, 384★)
daniel3303 runs as an external shell command, not a Claude Code plugin. This means **zero token overhead** — the statusline doesn't consume any context window. Worth investigating: could we offer a "lite mode" that runs externally?

### 2. TOML Configuration (from rz1989s + Wangnov)
Both 422★ and 187★ projects use TOML config files. Our env vars are simple but limited. A `~/.claude/statusline.toml` would allow:
```toml
[layout]
rows = ["basic+context", "turn+tools", "session", "daily"]
preset = "full"

[theme]
name = "ocean"

[display]
tools_count = 3
show_burn_rate = true
```

### 3. Rust Core (from Wangnov, 187★)
A Rust binary for the heavy computation (jq extraction, token calculation) could dramatically improve startup time. Our bash script does 19 jq calls — a Rust binary could do this in one pass.

### 4. Zero-Token External Mode (from daniel3303)
Key insight: running as an **external command** (not plugin) means statusline data doesn't count against the context window. daniel3303 achieves this with ~100 lines. We could offer both modes.

### 5. Rate Limit Priority Raised
3 of top 4 competitors show rate limits. This is now **the single most important missing feature**. Two approaches discovered:
- OAuth token from Keychain/credentials → `api.anthropic.com/api/oauth/usage`
- External cache file at `/tmp/claude/statusline-usage-cache.json` (60s cache)
